DGX Spark LLM Stack

A visual tour of how this stack runs locally on a single NVIDIA DGX Spark (GB10): the layers, the request flow, the deploy flow, and the hardware constraints that shape every design choice.

Section 1 · Stack overview

Three layers, one packet

Every request to this stack travels the same path: an OpenAI-compatible client hits LiteLLM on port 4000, which resolves the model alias and forwards to KubeAI on port 8000, which routes to the right vLLM pod for inference. The diagram below loops the round trip.

Section 2 · Hardware constraints

Why this stack looks the way it does

The GB10 is not a datacenter GPU. It is a single SoC with unified CPU+GPU memory, compute capability SM_121, and a fixed 128 GB pool. Every choice in this stack — the vLLM args, the image we pull, the eviction thresholds — exists because of one of the constraints below. None of these are tunable knobs. They are the rules.

unified memory

--gpu-memory-utilization=0.7

Higher values OOM-kill the pod. The 0.7 budget leaves 30% for the OS, the model loader, and KV cache headroom.

stability

--enforce-eager

Disables CUDA graph capture. Required on GB10 — graph capture deadlocks on unified memory in driver 590.x and is unstable on 580.x.

kv cache

--kv-cache-dtype=fp8

FP8 KV cache is the only way 256K context per request fits in the unified memory budget. Halves KV memory at near-zero quality cost.

arch

SM_121 vLLM image

Most upstream vLLM containers are compiled for SM_100. We pull scitrera/dgx-spark-vllm:0.17.0-t5 which is built for SM_121 and ships Transformers 5.x.

runtime

runtimeClassName: nvidia

K3s containerd default runtime does not see the GPU. KubeAI patches every model pod with the NVIDIA RuntimeClass via modelServerPods.jsonPatches.

shm

16 GiB /dev/shm

vLLM needs a real shared-memory mount for NCCL. Provided as an emptyDir medium: Memory volume sized to 16 GiB.

Section 3 · Configuration model

One source. Two consumers. Zero drift.

Every model in the stack is defined exactly once — in the models: block of ansible/group_vars/all/vars.yml. Two Jinja templates fan that single definition out into two K8s objects: a KubeAI Model CRD (lifecycle) and a LiteLLM ConfigMap (routing). Section 5 · Request flow (warm path)

A happy-path call to qwen35

Qwen 3.5 runs with min_replicas: 1 — it is always warm. This is what a typical request looks like end to end: alias resolution at LiteLLM, pod lookup at KubeAI, tokenization and generation at vLLM, response back up the stack. ~2.8s.

Section 6 · Request flow (scale-from-zero)

What a cold start looks like for qwen3

Qwen 3 runs with min_replicas: 0. Until the first request arrives, no pod exists. When KubeAI sees the request, it parks it briefly, scales the model up, waits for the pod to become Ready, then releases the request to the new pod. Total round trip ≈ 75 seconds. This is why frequently-used models live at min_replicas: 1.

Section 7 · Operational runbook

When something breaks, look here first

The five most common failure modes on this stack and exactly what to look at when they happen. Each card lists the symptom you'll see, the most likely cause, and the one or two commands that resolve or diagnose it.

symptom

Pod stuck in Pending after install

Likely cause: NVIDIA device plugin not running or containerd runtime not configured.

kubectl get nodes -o json | jq '.items[].status.allocatable["nvidia.com/gpu"]'
# null → re-run setup.sh --tags k3s,nvidia
symptom

Qwen 3 first request times out

Likely cause: Cold start exceeded the LiteLLM timeout. Expected behavior — see Section 6.

kubectl -n llm-stack get pods -l model=qwen3-30b-a3b
# Wait until Ready, then retry. Or set min_replicas: 1.
symptom

vLLM pod CrashLoopBackOff with SM_100 in log

Likely cause: Pulled a vLLM image not built for SM_121. Use a scitrera build.

kubectl -n llm-stack logs <pod> --previous | head -50
# Then update vllm_image in vars.yml and re-run --tags kubeai
symptom

LiteLLM 404 on a model name

Likely cause: ConfigMap rendered from stale vars.yml. The role auto-restarts but only on template change.

./setup.sh --tags litellm,validate
symptom

Zombie pods stuck in Unknown after reboot

Likely cause: Unclean shutdown. Controller-manager GC is set to 10 — should clear within seconds. If it doesn't:

scripts/recover-from-reboot.sh
# or: kubectl delete pod --force --grace-period=0 <name>
symptom

OOM kill on a vLLM pod under load

Likely cause: Someone tuned --gpu-memory-utilization above 0.7. Don't.

grep gpu_memory_utilization ansible/group_vars/all/vars.yml
# Must be 0.7 — see Section 2.
